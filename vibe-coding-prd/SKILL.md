# Vibe Coding PRD Skill

## 0. 这个 Skill 是干什么的

写一份**给 AI 直接执行**的 PRD，不是给团队 review 的传统 PRD。

| 维度 | 传统 PRD | Vibe Coding PRD |
|---|---|---|
| 读者 | 开发团队 | AI 编程助手（Claude Code / Cursor）|
| 模糊处 | 可以会议讨论补齐 | 必须明确，否则 AI 自由发挥 |
| 重点 | What + Why | How + What NOT to do + Acceptance |
| 形容词 | 允许（"现代化的"）| 禁止（必须用具体值）|
| 可执行性 | 给人理解 | 给 AI 直接落地 |

## 1. 触发逻辑

### 1.1 主动触发（立即启动，无需询问）

| 触发模式 | 例子 |
|---|---|
| 命令 | `/vibe-coding-prd`、`/vcprd` |
| 关键词组合 | "写 vibe coding PRD"、"给 Claude Code 用的 PRD"、"AI 能直接执行的 PRD" |
| 上下文转化 | 用户已聊了一段产品想法，然后说"把这些整理成 PRD" + 提到 vibe / AI / Claude Code |

→ 跳过询问，直接进入 Step 1。

### 1.2 被动建议（检测信号 + 询问）

#### 信号检测（同时满足 ≥3 个才触发询问）

| # | 信号 | 关键词举例 |
|---|---|---|
| ① | 描述**新项目** idea | "我想做"、"我在考虑做"、"我有个想法" |
| ② | 提到 **≥2 个功能** | "要能 A、还要 B"、"支持 C 和 D" |
| ③ | 提到**平台或技术栈** | "做成网页"、"用 Next.js"、"小程序"、"iOS app" |
| ④ | 提到**目标用户** | "给我自己用"、"给团队用"、"给客户用" |
| ⑤ | 问**下一步** | "接下来怎么办"、"我现在该做什么"、"怎么开始" |
| ⑥ | 当前目录**无 PRD.md** | 用 `ls docs/` 或类似检查 |

#### 询问话术（一次会话最多问 1 次）

```
💡 我注意到你在梳理一个新项目的思路（涉及 [识别到的功能/平台/用户]）。

要不要我帮你把这些整理成一份 vibe coding PRD？
（专门给 Claude Code / Cursor 直接执行的需求文档，4 步引导，约 15 分钟）

回 "好" 就开始 / 回 "不用" 继续聊 / 回 "稍后" 等你叫我
```

用户拒绝后，本会话不再建议（即使再次满足信号）。

### 1.3 不触发场景（避免误判）

- ❌ debug 现有代码 / 修 bug
- ❌ 修改已有 PRD（应该用"修订"流程，不是新建）
- ❌ 单纯技术咨询（"Next.js 怎么部署"）
- ❌ 需求评估"要不要做"（应触发 `brd`）
- ❌ 市场分析（应触发 `mrd`）
- ❌ 当前对话主线是 QA / 测试 / 部署
- ❌ 用户已在本会话拒绝过

## 2. 4 步工作流（每一步都必须用户确认才进下一步）

### Step 1: 前置信息确认

#### 1.1 扫描已有信息源

按优先级查找：
1. 当前目录下 `BRD.md` → 读取，提取核心信息
2. 当前目录下 `MRD.md` → 读取，提取核心信息
3. 当前目录下 `idea.md` / `notes.md` / `想法.md` → 读取
4. 对话历史中已有的项目描述

#### 1.2 缺失信息时问 5 个必答题

```
我准备帮你写 vibe coding PRD，需要先确认 5 件事：

① 一句话定位：你这个产品是什么？给谁用？（≤30 字）
   例：让我和三个 AI 角色一起反思决策的网页工具

② 平台：跑在哪里？（精确到框架）
   例：Next.js 14 + Vercel / SwiftUI 5 + iOS 17+ / 微信原生 + 云开发

③ 用户：谁会用？多少人？
   例：自用 1 人 / 团队 5 人 / 外部用户开放注册

④ 工期：多久内做完？
   例：1 天 / 1 周 / 1 个月

⑤ 参考产品（可选）：有没有类似的产品作为参考？
   例：类似 Notion AI / Cursor / 没有
```

#### 1.3 复述确认

```
✅ 我理解你的项目是这样：
- 产品：[复述]
- 平台：[复述]
- 用户：[复述]
- 工期：[复述]
- 参考：[复述]

理解对吗？回 "对" 进入下一步 / 回 "改 X" 我重新理解
```

**等用户明确回 "对" 才进入 Step 2**。任何修正都要重新复述确认。

- 部署：[具体]
- 第三方依赖：[列出 API / 服务，如 KIMI / Tavily / Supabase]

【v0.1 核心功能】（3-7 项）
1. [动词开头的用户行为]

【关键模块拆分】
- 模块 A：[职责]

- 模块 C：[职责]

【关键决策】
✅ 必做：[3-5 条]

🕐 暂缓：[1-3 条 + 放到哪个版本]

【风险点 / 不确定项】
- [列出需要用户拍板的产品决策]

**等用户回 "可以" / "改 X" 才进入 Step 3**。

按 C → S → P → A → G 顺序写。**每个模块写完先给用户看，等用户确认才写下一个模块**。

#### 3.1 模块 C — Context（上下文）

同上流程，写完等确认。

#### 3.3 模块 P — Specification（实现规范）

> ⚠️ 如果 P 模块超过 200 行，自动拆出 `docs/DESIGN.md`（含 P.1 + P.2）和 `docs/TECH.md`（含 P.3 + P.4），PRD.md 用 link 引用。

#### 3.4 模块 A — Acceptance（验收标准）

同上流程。

#### 3.5 模块 G — Guardrails（护栏）

同上流程。

### Step 4: 自查清单 + 二次确认

## 3. 5 个 PRD 模块详解（含 DO/DON'T 对照）

### 模块 C — Context（上下文）

**定义**：让 AI 知道"我在做什么项目、给谁用、跑在哪里"。

**为什么必须**：AI 没背景时按"通用最佳实践"做决策——把自用工具做成商业产品、把简单原型做成企业架构。

| ✅ DO | ❌ DON'T |
|---|---|
| 一句话定位（≤30 字） | 写愿景使命段落 |
| 平台精确到版本：`Next.js 14 + Vercel` | 写"web 应用" |
| 用户精确：`自用 1 人，不开放` | 写"目标用户分析" |
| 工期精确：`1 天内做完 v0.1` | 写"快速迭代" |

### 模块 S — Scope（范围）

**定义**：明确列出 必做 / 不做 / 暂缓 的功能边界。

**为什么必须**：AI 默认会过度发挥——你说做聊天，它给你加上语音/表情/历史搜索/分享。"不做"才是真护栏。

| ✅ DO | ❌ DON'T |
|---|---|
| 用版本号分层：v0.1 / v0.2 / v0.3 | 把所有想法堆一起 |
| MUST NOT 列表显式写："不做用户登录" | 写"暂时简单点" |
| 必做项用动词开头："用户能够 X" | 名词堆砌："X 功能" |
| 量化数量："3 个 Agent" | "多个" / "若干" |
| 每条不做 + 一句话原因 | 只写"不做 X" 不说为什么 |

### 模块 P — Specification（实现规范）

**定义**：对"怎么做"的精确描述，分 4 子节：

#### P.1 视觉规范

| ✅ DO | ❌ DON'T |
|---|---|
| 颜色用十六进制：`#fef3ef` | "温暖的米色" |
| 字体用具体字体名：`Newsreader, "PingFang SC", serif` | "优雅的衬线字体" |
| 间距用具体数值或 token：`24px` / `chat-gutter` | "适当间距" |
| 圆角分级：`pill / 12px / 8px / 4px` | "圆润的边角" |

#### P.2 行为规范

| ✅ DO | ❌ DON'T |
|---|---|
| 用「触发 → 响应」二元组："输入 `@` → 弹出选择浮层" | "支持 @ 提及" |
| 异常路径都列："API 失败 → 顶部 toast 红色 + 重试按钮" | "做好错误处理" |
| 时序明确："第 1 轮并行；第 2 轮起按 @" | "支持多轮对话" |

#### P.3 数据规范

| ✅ DO | ❌ DON'T |
|---|---|
| 用 TypeScript interface 表达：`interface Message { ... }` | "存储消息记录" |
| 存储 key 写死：`localStorage key = 'app-v1'` | "本地保存" |
| 版本号字段必填：`{ version: 1, ... }` | 不留升级空间 |

#### P.4 技术规范

| ✅ DO | ❌ DON'T |
|---|---|
| 框架版本精确：`Next.js 14 App Router` | "Next.js" |
| 用什么 + 不用什么："用 Tailwind，不用 shadcn / Antd" | "合适的 UI 方案" |
| 文件结构画出来：`app/, components/, lib/` | "标准的项目结构" |

### 模块 A — Acceptance（验收标准）

**定义**：每个功能怎么算"做对了"的可观测/可测试标准。

**为什么必须**：没有验收标准 = 做完了不知道对不对，AI 也不知道何时该停。

| ✅ DO | ❌ DON'T |
|---|---|
| 用 checkbox：`- [ ] 发消息后 3 个气泡同时流式输出` | "用户体验良好" |
| 每条可观测：截图对比 / 取色 / 字符串匹配 / 状态码 | "感觉自然" |
| 含异常路径：`- [ ] KIMI 401 时显示 toast` | 只写正常路径 |
| 含跨平台：`- [ ] iOS Safari 与 Chrome 一致` | 只测一个浏览器 |

### 模块 G — Guardrails（护栏）

**定义**：禁止做的事 + 必须问用户的决策时机。

**为什么必须**：AI 会做"善意但越权"的事——顺手重构、加依赖、改架构；显式禁止才能止损。

| ✅ DO | ❌ DON'T |
|---|---|
| 每条禁令配一句"为什么"：`❌ 不要换模型 / 原因：用户已购 key` | "尽量保持模型一致" |
| 禁令用否定句开头 | 委婉表达"建议不要" |
| 含「必问 / 自决」决策时机表 | 一律问 / 一律自决 |

**G 模块必含的 4 类禁令**：
1. **不要换技术栈**（已选定的框架/库/服务）
2. **不要顺手重构**（修 bug 时不改其他代码）
3. **不要加未确认的依赖**（npm install 前必问）
4. **不要在 .env / 代码 / 截图中暴露 key**

**决策时机表（必含）**：

| 场景 | 必问用户 / 可自决 |
|---|---|
| 改产品颜色 / 字体 / 角色性格 | 必问 |
| 改文件名 / 变量名 | 自决 |
| 加 npm 依赖 | 必问 |
| 修 bug 的实现细节 | 自决 |
| 改架构 / 数据结构 | 必问 |
| 改 API 接口签名 | 必问 |

## 6. 输出文件结构（解耦设计）

```

### 自动拆分规则（任何模块 > 200 行时）

```
docs/
├── PRD.md            # 主文档：保留 C / S / A / G + link 引用拆出去的模块
├── DESIGN.md         # P.1 视觉 + P.2 行为

- DESIGN.md / TECH.md 里**只放规范，不放需求**（需求永远在 PRD.md）

- 每次大改前自动备份到 `prd-archive/PRD-v{X.Y}-{date}.md`

## 8. 隐性规则（执行时必须遵守）

### 规则 A：失焦警报
执行 PRD 时如发现任何字段模糊或相互矛盾 → **立即停下问用户**，不要自己脑补。
本规则在 Guardrails 模块自动写入。

### 规则 B：PRD 是契约 + 日志

4 步流程的每一步**都必须等用户回 "好"/"对"/"可以" 才进下一步**，不要默默推进。

## 10. 错误处理

### 用户在 Step N 中途说"算了不写了"
- 回应：保留已写入的部分（如 `docs/PRD.md` 已有 C 模块），不删除
- 提示：`已保留进度。下次回 /vcprd 可继续，会从未完成的模块开始`

### 用户在多轮中改了主意（如 Step 3 中说"前面 Scope 有问题"）
- 回应：跳回到 Step 2（架构）重新确认，再回 Step 3 重写受影响模块
- 不要直接改 PRD 文件，先和用户对齐改动范围

### docs/PRD.md 已存在
- 不直接覆盖
- 询问：`docs/PRD.md 已存在。要 (1) 备份后重写 (2) 在原文档上修订 (3) 取消？`
- 选 (1) → 移到 `docs/prd-archive/PRD-{old-version}-{date}.md` 后重写
- 选 (2) → 进入"修订模式"（保留原结构，逐节比对）
- 选 (3) → 退出 skill
